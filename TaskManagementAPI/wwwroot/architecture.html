Architecture Overview

Understanding the design philosophy and inner workings of the Task Management API.

The Task Management API is built using Modular Monolith and Clean Architecture principles. Features are logically separated into independent modules while sharing a common infrastructure, making the system highly testable, maintainable, and ready for future decomposition into microservices if needed.

Architectural Goal
Each domain (Users, Projects, Tasks, Notifications) is fully encapsulated — teams can work independently on different modules without risk of cross-module breakage.

Modular Structure

Four modules, each with a hard boundary: its own DbContext, its own EF Core migration history, and its own DI registration. No shared database joins across module lines.

Shell 
Modules/
├── Users/
│   ├── Domain/            Entity: ApplicationUser (IdentityUser)
│   ├── Application/       Services: AuthService, UserService
│   ├── Infrastructure/    UsersDbContext  ←  own migration history
│   └── Presentation/      Controllers: AuthController, UsersController
│
├── Projects/
│   ├── Domain/            Entities: Project, ProjectMember
│   ├── Application/       Services: ProjectService
│   ├── Infrastructure/    ProjectsDbContext  ←  own migration history
│   └── Presentation/      Controllers: ProjectsController
│
├── Tasks/
│   ├── Domain/            Entities: TaskItem, TaskDependency, TimeEntry
│   ├── Application/       Services: TaskService
│   ├── Infrastructure/    TasksDbContext  ←  own migration history
│   └── Presentation/      Controllers: TasksController
│
└── Notifications/
    ├── Domain/            Entity: Notification
    ├── Application/       Services: NotificationService
    ├── Infrastructure/    NotificationsDbContext  ←  own migration history
    └── Presentation/      Hubs: NotificationsHub
Module Communication Rule
When the Tasks module needs a user's display name it calls IUserService — defined in Tasks/Application, implemented in Users/Infrastructure. It never reaches into UsersDbContext directly. This is the hard rule that makes the monolith modular.

Clean Architecture Layers

Each module follows a four-layer structure to keep concerns fully separated.

Presentation Layer

Handles API entry points, request routing, and response formatting. Controllers stay intentionally thin.

  • REST Controllers
  • Middleware & auth filters
  • Swagger/OpenAPI spec

Application Layer

Business orchestration — services, use-case handlers, validation, and cross-module event dispatch.

  • Service & use-case classes
  • DTOs & request models
  • FluentValidation rules

Infrastructure Layer

All I/O concerns: database persistence, caching, email, file storage, and third-party integrations.

  • Module-scoped DbContexts
  • EF Core entity configs
  • Redis cache providers

Domain Layer

The pure, framework-free heart of each module. Contains entities, value objects, and domain events.

  • Aggregate roots & entities
  • Domain events
  • Value objects & exceptions
Dependency Direction
Dependencies only flow inward: Presentation → Application → Domain. Infrastructure implements interfaces defined in Application, so Domain has zero framework dependencies.

Request Pipeline

Every HTTP request passes through an ordered middleware stack before reaching a controller. Understanding this order is critical for debugging and extending the API.

① HTTPS Redirection

All plain HTTP connections are redirected to HTTPS immediately. In the Replit development environment this is suppressed so the proxied preview iframe works without cert issues.

② Static Files

The wwwroot/ directory is served before any routing logic. Documentation pages, stylesheets, and scripts are delivered without touching the ASP.NET Core controller pipeline.

③ Structured Request Logging

Serilog's request-logging middleware captures method, path, status code, and elapsed time. Every log line carries a RequestId that threads through the entire downstream pipeline.

④ Rate Limiting

A sliding-window limiter checks request budget before any authentication occurs. Clients that exceed 100 req/min receive an immediate 429 — the identity layer is never reached.

⑤ Authentication & Authorization

JWT bearer token is validated. The [Authorize] attribute enforces role requirements. Unauthenticated requests stop here with a 401; forbidden roles stop with 403.

⑥ Response Compression

Gzip and Brotli compression is applied on the way out for all JSON responses larger than 1 KB. Typically reduces API payload sizes by 60–80% on list endpoints.

⑦ FluentValidation

Request DTOs are validated before the controller action body executes. Invalid requests return a 400 with a structured error map — the service layer never sees bad input.

⑧ SignalR & Controller Routing

Requests are dispatched either to the /hubs/notifications SignalR hub or to the appropriate module controller. Controllers delegate to service classes and return structured responses.

Adding custom middleware
Register middleware in Program.cs between the appropriate steps above. If your middleware needs access to the authenticated user, place it after UseAuthentication(). If it should run regardless of auth (e.g. request telemetry), place it before.

Design Patterns

Proven patterns applied consistently across every module.

Service Layer Pattern

Controllers delegate all logic to dedicated service classes. Controllers remain thin — they handle HTTP routing, model binding, and response shaping only. Services handle validation, authorization checks, and domain orchestration.

Repository Pattern

All data access goes through repository interfaces defined in the Application layer and implemented in Infrastructure. The Application layer never touches DbContext directly — this makes unit testing trivial with in-memory or mock repositories.

Modular DbContexts

Each module has its own DbContext and its own EF Core migration history. This enforces hard data boundaries between modules. If a module needs data from another, it goes through that module's service — never by joining across contexts directly.

No Cross-context Queries
Cross-module data fetching is always mediated through the owning module's service interface, never via raw SQL or shared DbContext. This is intentional — it preserves module independence.

Technology Stack

Carefully chosen, battle-tested technologies forming a coherent platform.

.NET 9 / ASP.NET Core

Core framework for high-performance API development with minimal overhead and first-class async support.

PostgreSQL + EF Core

Robust relational database via Npgsql driver. EF Core 9 handles migrations, compiled queries, and owned entities.

Redis

In-memory distributed cache for sub-millisecond reads on hot data. Supports cache-aside and write-through patterns.

SignalR

Real-time bidirectional communication hub for instant event delivery to all connected clients.

FluentValidation

Rich, expressive validation rules decoupled from request models — easily unit-tested in isolation.

Serilog

Structured, sink-agnostic logging. Production logs can be routed to file, Seq, Elastic, or any structured sink.
